Topic 4: **SaaS Tiering & Monetization Engine**.
This module transforms the application from a utility tool into a scalable business by defining plan limits, enforcing feature access, and automating subscription billing via Dodo Payments.

---

### **Task 1: Update User Model with Subscription Fields**
- **suggested-assignee**: Backend
- **priority**: High
- **consideration**: Use an Enum or string constants (`starter`, `pro`, `studio`) to avoid magic numbers. 
- **description**: We need to track which tier a user is on and when their subscription expires to enforce access control.
- **requirements**:
    1. Add `SubscriptionTier` (string) to `models.User`.
    2. Add `SubscriptionID` (string) to store the Dodo identifier.
    3. Add `SubscriptionExpiresAt` (time) field.
- **acceptance-criteria**:
    - Default users are created with the `starter` tier.
    - Database schema reflects the new fields.

### **Task 2: Define Tier Configuration in Backend**
- **suggested-assignee**: Senior
- **priority**: High
- **consideration**: Hardcoding these in the service is okay for V1, but eventually, these should be in a config file or DB.
- **description**: Create a centralized map or struct that defines the limits (Storage, Projects, Fees) for each tier.
- **requirements**:
    1. Define `StarterTier`: 5GB Storage, 2 Projects, 5% Fee.
    2. Define `ProTier`: 100GB Storage, Unlimited Projects, 2% Fee.
    3. Define `StudioTier`: 500GB Storage, Unlimited Projects, 1% Fee.
- **acceptance-criteria**:
    - A utility function `GetTierConfig(tierName)` returns the correct struct.

### **Task 3: Feature-Gating Middleware logic**
- **suggested-assignee**: Backend
- **priority**: High
- **consideration**: This middleware should return a `402 Payment Required` status code if a limit is hit, which the frontend can intercept.
- **description**: Implement a reusable logic layer to check if a user is exceeding their tier's limits before allowing an action.
- **requirements**:
    1. Create a service method `CanCreateProject(userID)`.
    2. Create a service method `HasStorageSpace(userID, bytesToAdd)`.
- **acceptance-criteria**:

### **Task 4: Dynamic Platform Fee Deduction**
- **suggested-assignee**: Backend
- **priority**: High
- **consideration**: Ensure the calculation always rounds in favor of the platform (upward for fees) to avoid precision loss.
- **description**: Update the escrow release logic to pull the fee percentage dynamically based on the creator's subscription tier.
- **requirements**:
    1. Modify `internal/services/escrow.go`.
    2. Look up the creator's `SubscriptionTier` during the `ReleaseMilestone` call.
    3. Use the corresponding fee percentage from the Tier Config.
- **acceptance-criteria**:
    - A Pro user releasing a $100 milestone is charged $2, while a Starter user is charged $5.
