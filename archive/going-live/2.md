Topic 2: **Escrow & Milestone Logic Hardening**. 
This module focuses on financial safety, preventing fraud, and automating the transitions between "In Progress," "Submitted," and "Completed."

---

### **Task 1: Proposal Math Validation (Server-side)**
- **suggested-assignee**: Backend
- **priority**: High
- **consideration**: Clients often verify the "Total" but don't add up the milestones. A mismatch is a major fraud risk.
- **description**: Implement validation logic to ensure the sum of all milestone `amount_cents` matches the proposal's `total_amount`.
- **requirements**:
    1. Update `internal/services/proposal.go`.
    2. In the `CreateVersion` method, loop through milestones and sum their amounts.
    3. Return a `400 Bad Request` if the sum does not equal the version total.
- **acceptance-criteria**:
    - An API request with milestones totaling 10,000 but a total of 12,000 is rejected with a clear error message.

---

### **Task 2: Enforce Escrow for "None" Type Milestones**
- **suggested-assignee**: Backend
- **priority**: High
- **consideration**: Currently, "none" type milestones (deposits) are treated as direct payments. They should still be "held" until the project is officially started to protect the client.
- **description**: Change the logic so that `deliverable_type: 'none'` milestones are held in Zeyaroo’s ledger until the project status moves to `active`.
- **requirements**:
    1. Modify `internal/services/escrow.go`.
    2. Change the flow for `none` type so it doesn't bypass the `held` state in the transaction ledger.
- **acceptance-criteria**:
    - A deposit milestone shows as "Funded (Held)" in the database instead of immediately transferring to the creator's balance.

---

### **Task 3: Implementation of Database Transactions for Fund Release**
- **suggested-assignee**: Senior
- **priority**: High
- **consideration**: If the milestone update succeeds but the creator's balance update fails, you lose money.
- **description**: Wrap the `ReleaseMilestone` logic in a SQL transaction (Atomicity).
- **requirements**:
    1. Update `internal/services/escrow.go`.
    2. Use `s.db.Begin()` at the start of the release process.
    3. Rollback on any error; commit only if both the status update and the balance credit succeed.
- **acceptance-criteria**:
    - If a network error occurs mid-payout, the milestone status remains "Submitted" and the creator's balance remains unchanged.

---

### **Task 4: 7-Day Auto-Release Background Job**
- **suggested-assignee**: Backend
- **priority**: Medium
- **consideration**: Prevents "Ghost Clients" from trapping creator funds indefinitely.
- **description**: Create a cron-like task that automatically releases funds for milestones that have been in "Submitted" status for more than 7 days without a dispute.
- **requirements**:
    1. Create a new worker function in `internal/services/cleanup.go`.
    2. Query: `status = 'submitted' AND completed_at < now - 7 days`.
    3. Trigger the `ReleaseMilestone` logic for each result.
- **acceptance-criteria**:
    - A milestone submitted 8 days ago is automatically marked "Completed" when the task runs.

### **Task 5: Client "Quality Confirmation" Checkbox**
- **suggested-assignee**: Frontend
- **priority**: Medium
- **consideration**: This is a legal "Safe Harbor" feature.
- **description**: Add a mandatory checkbox to the "Release Payment" modal in the UI.
- **requirements**:
    1. Edit `MilestoneList.tsx`.
    2. The "Approve & Release" button must be disabled until the user checks: *"I have reviewed the work and confirm it meets the agreed standards."*
- **acceptance-criteria**:
    - Client cannot release funds without explicitly ticking the confirmation box.

### **Task 6: Project Deletion Lock on Funded Milestones**
- **suggested-assignee**: Backend
- **priority**: High
- **consideration**: Creators shouldn't be able to delete a project if Zeyaroo is currently holding client money for it.
- **description**: Prevent project deletion if any milestone has the status `funded` or `submitted`.
- **requirements**:
    1. Update `internal/services/project.go`.
    2. In `DeleteProject`, check the status of all associated milestones.
- **acceptance-criteria**:
    - API returns `403 Forbidden` if a user tries to delete a project with "Held" funds.

---

### **Task 7: Milestone Sequence Enforcement**
- **suggested-assignee**: Backend
- **priority**: Medium
- **consideration**: Prevents clients from paying the last milestone before the first one (which can mess up delivery logic).
- **description**: Ensure milestones are funded in their numerical sequence.
- **requirements**:
    1. Update `internal/services/escrow.go`.
    2. In `FundMilestone`, check if all previous milestones (`sequence < current`) are already `funded` or `completed`.
- **acceptance-criteria**:
    - Client gets an error if they try to pay Milestone #3 while Milestone #2 is still "Pending."

---

### **Task 8: Admin Dispute Resolution API**
- **suggested-assignee**: Backend
- **priority**: High
- **consideration**: You need a way to break an "Escrow Deadlock" manually.
- **description**: Create an internal-only endpoint to force-resolve a disputed milestone.
- **requirements**:
    1. New endpoint `POST /admin/milestones/:id/resolve`.
    2. Accepts `action: 'release'` or `action: 'refund'`.
- **acceptance-criteria**:
    - An admin call can move a milestone from `disputed` to `completed` (release) or `pending` (refund).

### **Task 9: Request Changes - Reason History**
- **suggested-assignee**: Fullstack
- **priority**: Low
- **consideration**: Useful for resolving future disputes if the client keeps changing their mind.
- **description**: Instead of overwriting the `changes_reason`, store a history of change requests.
- **requirements**:
    1. Create `models.MilestoneFeedback`.
    2. Store `milestone_id`, `author_id`, `body`, and `created_at`.
    3. Update the UI to show this as a simple thread.
- **acceptance-criteria**:
    - Creator can see a list of all previous feedback provided by the client for that milestone.

---

### **Task 10: Automatic Delivery Lock on "Submitted"**
- **suggested-assignee**: Backend
- **priority**: Medium
- **consideration**: Creators shouldn't be able to remove files once they've claimed the milestone is "Done."
- **description**: When a milestone moves to `submitted`, the associated `DeliveryAlbum` must be locked.
- **requirements**:
    1. In `internal/services/escrow.go`, call `albumService.Lock(albumID)` when `CompleteMilestone` (Submit) is called.
- **acceptance-criteria**:
    - Any attempt to `DeleteMedia` or `RemoveFromAlbum` for files in a locked delivery album fails.

---

### **Task 11: Refund Logic Stub for Canceled Projects**
- **suggested-assignee**: Backend
- **priority**: Medium
- **consideration**: If a project is canceled, held funds must return to the client.
- **description**: Implement the logic to reverse an escrow hold.
- **requirements**:
    1. Create `RefundEscrow(milestoneID)` in the escrow service.
    2. Update the `EscrowTransaction` status to `refunded`.
- **acceptance-criteria**:
    - Calling refund creates a "Negative" ledger entry for the creator's pending balance and a "Success" state for the refund record.
