Topic 9: **Database & DevOps Hardening**.
This module is about moving from a "Project" to a "Platform." It ensures the application is stable under high concurrent traffic, secure from infrastructure-level attacks, and efficient to deploy and scale globally.

---

### **Task 1: Implement Support for PostgreSQL**
- **suggested-assignee**: Backend
- **priority**: High
- **consideration**: GORM handles multiple drivers well, but PostgreSQL is required for concurrent writes (SQLite locks the whole file). 
- **description**: Update the database initialization to support PostgreSQL based on an environment variable.
- **requirements**:
    1. Update `internal/database/database.go`.
    2. Add logic to check `DB_DRIVER` env var.
    3. Import `gorm.io/driver/postgres` and configure connection string parsing.
- **acceptance-criteria**:
    - App starts and runs `AutoMigrate` successfully when connected to a local Postgres instance.

### **Task 2: Production Docker Compose Configuration**
- **suggested-assignee**: Senior
- **priority**: High
- **consideration**: Ensure the Postgres container uses a persistent volume so data isn't lost on restart.
- **description**: Create a `docker-compose.prod.yml` file that includes the API, UI, and a managed PostgreSQL database.
- **requirements**:
    1. Define `db` service using `postgres:16-alpine`.
    2. Set up health checks to ensure the API only starts after the DB is ready.
    3. Configure standard production networks and volumes.
- **acceptance-criteria**:
    - Running `docker-compose up` spins up a fully functional environment with a real database.

### **Task 3: Optimize Frontend Multi-Stage Build**
- **suggested-assignee**: Junior
- **priority**: Medium
- **consideration**: The current image is likely carrying unnecessary build artifacts.
- **description**: Hardening the `phto-ui/Dockerfile` to ensure the final image is as small as possible.
- **requirements**:
    1. Ensure `node_modules` are NOT copied into the final Nginx stage.
    2. Use `nginx:stable-alpine` for the final stage.
    3. Implement a custom `nginx.conf` that handles SPA routing (`try_files $uri /index.html`).
- **acceptance-criteria**:
    - Final UI image size is under 50MB.

### **Task 4: Implement Structured Logging (slog)**
- **suggested-assignee**: Backend
- **priority**: Medium
- **consideration**: Plain text logs are hard to search in production. JSON logs are standard for Logstash/CloudWatch.
- **description**: Replace standard `fmt.Printf` or `log` calls with Go 1.21's structured logging (`slog`).
- **requirements**:
    1. Initialize `slog` with a JSON handler in `main.go` if `ENV=production`.
    2. Update service layers to include context (e.g., `user_id`, `project_id`) in log attributes.
- **acceptance-criteria**:
    - Application output is valid JSON in the console.

### **Task 5: Database Connection Pooling Configuration**
- **suggested-assignee**: Senior
- **priority**: Medium
- **consideration**: Prevents "Too many connections" errors during traffic spikes.
- **description**: Explicitly configure GORM's underlying `sql.DB` connection pool settings.
- **requirements**:
    1. Set `MaxIdleConns` (suggest 10).
    2. Set `MaxOpenConns` (suggest 100).
    3. Set `ConnMaxLifetime` (suggest 1 hour).
- **acceptance-criteria**:
    - Connection pool settings are visible/configurable via environment variables.

### **Task 6: Hardening Docker Containers (Non-Root User)**
- **suggested-assignee**: Junior
- **priority**: Medium
- **consideration**: Running as root in Docker is a security risk if the container is compromised.
- **description**: Update the API Dockerfile to use a non-privileged user.
- **requirements**:
    1. Use `RUN adduser -D zeyaroo` in the Dockerfile.
    2. Ensure the binary has permissions to write to the `/app/uploads` directory.
    3. Switch to `USER zeyaroo`.
- **acceptance-criteria**:
    - `docker exec zeyaroo-api whoami` returns "zeyaroo" instead of "root".

### **Task 7: Implement API Rate Limiting Middleware**
- **suggested-assignee**: Backend
- **priority**: High
- **consideration**: Protects the app from brute-force attacks on OTP endpoints and scraping of creator profiles.
- **description**: Add a simple rate-limiter to the Gin router.
- **requirements**:
    1. Use a library like `github.com/ulule/limiter`.
    2. Set a limit of 60 requests per minute for public endpoints.
    3. Set a stricter limit (5 per 10 min) for `request-otp`.
- **acceptance-criteria**:
    - Exceeding limits returns a `429 Too Many Requests` status code.

### **Task 8: Centralized Secrets Management Strategy**
- **suggested-assignee**: Senior

- **consideration**: Don't check `.env` into git. Use a `.env.example`.
- **description**: Clean up the current env loading and ensure secrets are never logged.

- **suggested-assignee**: Junior

- **description**: A simple shell script to dump the Postgres DB and upload it to a private Backblaze bucket.

### **Task 11: Production Health Check Endpoints**
- **suggested-assignee**: Junior

### **Task 12: Graceful Shutdown Implementation**
- **suggested-assignee**: Backend
- **priority**: Low
- **consideration**: Prevents data corruption by allowing the server to finish processing current requests before stopping.
- **description**: Listen for termination signals (SIGINT, SIGTERM) in the main Go process.
- **requirements**:
    1. Wrap the Gin server in an `http.Server` struct.
    2. Use a channel to catch OS signals.
    3. Call `server.Shutdown(ctx)` and allow 5 seconds for cleanup.
- **acceptance-criteria**:
    - Stopping the container logs "Shutting down..." and waits for active requests to finish.
