Topic 6: **Automated Retention & CRM**. 
This module focuses on keeping users engaged and converting leads by automating communication and turning Zeyaroo into the creator’s central "Source of Truth" for project details.

---

### **Task 1: Background Scheduler Infrastructure**
- **suggested-assignee**: Backend
- **priority**: High
- **consideration**: Use a simple Go ticker or a library like `gocron`. Ensure it doesn't run multiple times if you have multiple server instances (use a basic DB lock if needed).
- **description**: We need a reliable way to run recurring checks (e.g., every hour) to find proposals or milestones that need a nudge.
- **requirements**:
    1. Implement a worker loop in `cmd/server/main.go`.
    2. Create a registry for "Job" functions.
- **acceptance-criteria**:
    - Logs show the scheduler running every hour without crashing the main API.

---

### **Task 2: WhatsApp API Integration Service**
- **suggested-assignee**: Backend
- **priority**: High
- **consideration**: For the Sri Lankan market, use a provider like AISensy or Twilio. Templates must be pre-approved by Meta.
- **description**: Implement a service to send WhatsApp messages based on system events.
- **requirements**:
    1. Create `internal/services/whatsapp.go`.
    2. Implement `SendMessage(to, templateName, variables)`.
    3. Add `whatsapp_number` to the User model.
- **acceptance-criteria**:
    - A test script successfully triggers a WhatsApp message to a real phone number.

---

### **Task 3: Automated "New Inquiry" WhatsApp Alert**
- **suggested-assignee**: Backend
- **priority**: High
- **consideration**: Speed is everything. The message should contain a "Quick Link" to the inquiry.
- **description**: Send an immediate WhatsApp notification to the creator when a new inquiry is received via their public profile.
- **requirements**:
    1. Hook into the `CreateInquiry` service.
    2. Trigger WhatsApp if the creator has a valid number saved.
- **acceptance-criteria**:
    - Creating an inquiry via the public page results in a WhatsApp notification to the creator within 60 seconds.

---

### **Task 4: Automated "Unaccepted Proposal" Reminders**
- **suggested-assignee**: Backend
- **priority**: Medium
- **consideration**: Don't spam. Send only one reminder 48 hours after the proposal was sent.
- **description**: If a client has viewed a proposal but hasn't accepted it, send a nudge email/WhatsApp to keep the deal moving.
- **requirements**:
    1. Worker logic: Find `proposals` where `status = 'negotiating'` and `updated_at` is 48h old.
    2. Trigger the `ProposalReminder` notification template.
- **acceptance-criteria**:
    - Test case: A proposal from 2 days ago receives a reminder; a proposal from 1 hour ago does not.

---

### **Task 5: "Milestone Funded" Notification for Creators**
- **suggested-assignee**: Junior
- **priority**: Medium
- **consideration**: This is the "Dopamine Hit" for the creator. Make it celebratory.
- **description**: Notify the creator immediately when a client pays a milestone.
- **requirements**:
    1. Hook into the DodoPayments/LKR webhook success handler.
    2. Send Email + WhatsApp: "Great news! [Client Name] just funded your milestone [Title] for [Amount]."
- **acceptance-criteria**:
    - Creator receives a notification immediately after a successful checkout.

---

### **Task 6: Project "Intake Questionnaire" Model**
- **suggested-assignee**: Backend
- **priority**: Medium
- **consideration**: Keep it flexible. Use a JSONB field (if on Postgres) or a JSON string (if on SQLite) to store form data.
- **description**: Allow creators to attach a "Questionnaire" to a proposal to collect project details (e.g., "What is the venue address?").
- **requirements**:
    1. Create `models.Questionnaire`.
    2. Fields: `project_id`, `schema` (JSON definition of questions), `responses` (JSON results).
- **acceptance-criteria**:
    - Database can store a list of questions and their corresponding answers for a specific project.

### **Task 7: Simple Questionnaire Builder UI**
- **suggested-assignee**: Frontend
- **priority**: Medium
- **consideration**: For V1, don't build a drag-and-drop builder. Just a "Add Row" list of text inputs.
- **description**: UI for creators to define the questions they want to ask the client during the proposal phase.
- **requirements**:
    1. Add a "Questionnaire" tab to the Proposal creation flow.
    2. Allow adding "Text" or "Multiple Choice" questions.
- **acceptance-criteria**:
    - Creator can save a list of questions associated with a proposal.

### **Task 8: Client-Side Questionnaire Response View**
- **suggested-assignee**: Frontend
- **priority**: High
- **consideration**: The client should ideally answer these *before* they release the first payment.
- **description**: Display the questionnaire to the client on the Public Proposal page.
- **requirements**:
    1. Render the questions defined by the creator.
    2. Save responses to the backend via a new endpoint.
- **acceptance-criteria**:
    - A guest client can fill out the form and hit "Save Info" successfully.

### **Task 9: "Post-Delivery" Feedback Loop (7 Days After)**
- **suggested-assignee**: Backend
- **priority**: Low
- **consideration**: Use this to generate testimonials or Google Review prompts.
- **description**: One week after the final milestone is released, email the client asking for a review.
- **requirements**:

- **acceptance-criteria**:

### **Task 10: Notification Preferences Settings**
- **suggested-assignee**: Fullstack
- **priority**: Medium
- **consideration**: Essential for compliance (allowing users to opt-out of WhatsApp).
- **description**: Let users toggle which notifications they want to receive and through which channel.
- **requirements**:
    1. Add `NotificationSettings` (JSON) to User model.
    2. UI in Settings: Checkboxes for [Inquiry, Payment, Reminder] x [Email, WhatsApp].
- **acceptance-criteria**:
    - Users can disable WhatsApp notifications while keeping Email active.
