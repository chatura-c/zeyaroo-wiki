This is Topic 1: Global & Local Payment Infrastructure. The goal here is to move from a mock environment to a production-ready, hybrid system that handles international USD (DodoPayments) and local LKR (Manual/PayHere) flows.

Task 1: Extend Milestone and Project Models for Multi-Currency

suggested-assignee: Backend

priority: High

consideration: We must use ISO 4217 codes (USD, LKR). Defaulting to LKR for now is fine, but the DB must support variation.

description: Currently, the system assumes a single currency context. We need to add currency_code fields to both the Project and individual Milestones to support the Payment Router.

requirements:

Add CurrencyCode (string) to models.Project.

Add CurrencyCode (string) to models.Milestone.

Run a migration to update the SQLite/Postgres schema.

acceptance-criteria:

New projects can be created with a specific currency.

Milestones inherited the project's currency by default.

Task 2: Create Payment Gateway Strategy Interface

suggested-assignee: Senior

priority: High

consideration: This allows us to swap Dodo for Stripe or PayHere without changing business logic in the escrow_service.

description: Implement a Go interface that abstracts the "Fund" and "Payout" actions.

requirements:

Define Gateway interface in internal/gateways/gateway.go.

Methods needed: CreateCheckout(amount, currency, metadata) (URL, error) and TriggerPayout(amount, currency, destination) (error).

acceptance-criteria:

The existing mock_gateway.go implements this new interface.

Task 3: Implement DodoPayments Checkout Service

suggested-assignee: Backend

priority: High

consideration: Use the metadata field in Dodo to store the milestone_id. This is the only way to link the payment back to our DB.

description: Create the concrete implementation for DodoPayments to generate checkout links for international clients.

requirements:

Create internal/gateways/dodopayments.go.

Integrate with Dodo’s /v1/checkouts endpoint.

Return the checkout_url to the handler.

acceptance-criteria:

Calling the service generates a real Dodo checkout link in the logs or response.

Task 6: Implement Regional Gateway Resolver Logic

suggested-assignee: Senior

priority: High

consideration: This is the "brain" of the payment system. If Project=LKR, use local logic. If Project=USD, use Dodo.

description: Logic in the service layer to decide which gateway to initialize based on milestone data.

requirements:

Create internal/gateways/router.go.

Logic: if milestone.Currency == "LKR" { return localGateway } else { return dodoGateway }.

acceptance-criteria:

When a client clicks "Pay," the correct gateway is triggered based on the project's currency.

suggested-assignee: Backend

priority: High

Verify the Dodo signature.

Extract milestone_id from metadata and update milestone to funded.

Task 10: Platform Fee Deduction Logic

suggested-assignee: Senior

priority: High

consideration: This is how the business survives. Fees should be configurable.

description: When calculating the payout to the creator, subtract the Zeyaroo commission.

requirements:

Add PlatformFeePercentage to the config.

In internal/services/escrow.go, calculate payout = total - (total * fee).

Store the fee amount in the EscrowTransaction record for accounting.

acceptance-criteria:

If fee is 5%, a $100 payment results in a $95 recorded payout for the creator.

Task 11: Transaction Ledger Table for Auditing

suggested-assignee: Backend

description: Create a centralized table to record every single movement of money.

requirements:

Create models.TransactionLedger.

acceptance-criteria:

Every successful checkout and payout creates a corresponding row in this table.
