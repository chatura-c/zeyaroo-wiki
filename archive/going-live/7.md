Topic 7: **Media Processing & Storage Optimization**. 
This module is critical for the "Cost-to-Serve" strategy. We aim to move the heavy lifting of media (bytes and processing) off your Go server and directly into the cloud (Backblaze B2 + Cloudflare), while automating thumbnails and watermarks.

---

### **Task 1: Implement S3-Compatible Pre-signed Uploads (Backend)**
- **suggested-assignee**: Backend
- **priority**: High
- **consideration**: This stops files from passing through your Go server, saving RAM and CPU. Use the `aws-sdk-go-v2` S3 package.
- **description**: Create an endpoint that generates a temporary URL allowing the frontend to upload directly to Backblaze B2.
- **requirements**:
    1. New endpoint `POST /api/v1/projects/:id/media/upload-url`.
    2. Logic: Generate a `PutObject` pre-signed URL with a 15-minute expiry.
    3. Return the `URL` and the `StorageKey` to the frontend.
- **acceptance-criteria**:
    - The API returns a valid URL that allows a `PUT` request without an `Authorization` header.

---

### **Task 2: Transition UI to Direct-to-Cloud Uploads**
- **suggested-assignee**: Frontend
- **priority**: High
- **consideration**: You need to update the upload progress bar to work with the new `PUT` request logic.
- **description**: Refactor the upload logic in `ProjectDetail.tsx` to use the pre-signed URL.
- **requirements**:
    1. Call the `upload-url` API to get a destination.
    2. Use `axios.put(url, file)` to send the data directly to B2.
    3. Once successful, call a "Commit" endpoint to notify the backend the file is ready.
- **acceptance-criteria**:
    - Files appear in the project after being uploaded directly to B2.

---

### **Task 3: Refactor Storage Path Hierarchy (Prefixes)**
- **suggested-assignee**: Backend
- **priority**: Medium
- **consideration**: Consistent paths make "Delete User" or "Calculate Project Size" operations simple S3 prefix queries.
- **description**: Standardize the B2 object keys using the user-centric hierarchy.
- **requirements**:
    1. Key format: `users/{user_id}/projects/{project_id}/media/{media_id}/{filename}`.
    2. Update the `s3.go` storage driver to use this logic for all new uploads.
- **acceptance-criteria**:
    - Files are organized in the B2 bucket under clear, predictable prefixes.

---

### **Task 4: Background Thumbnail Generation Infrastructure**
- **suggested-assignee**: Senior
- **priority**: High
- **consideration**: Do not process images in the main request-response cycle. Use a Go `chan` or an internal worker pool.
- **description**: Set up a background worker that triggers whenever a new media item is "Committed."
- **requirements**:
    1. Implement a `MediaProcessor` worker.
    2. Download the `orig` file, resize it using `disintegration/imaging` or `bimg`.
    3. Upload the resulting `_thumb.jpg` to the same prefix.
- **acceptance-criteria**:
    - Uploading a 20MB image results in a small thumbnail appearing in B2 within 10 seconds.

### **Task 5: Automated Watermarking Service**
- **suggested-assignee**: Backend
- **priority**: High
- **consideration**: This is the core "Client-Security" feature. Watermarks should be semi-transparent and placed in the center.
- **description**: Create a service that applies a "Zeyaroo" or "Creator Custom" watermark to images.
- **requirements**:
    1. Function `ApplyWatermark(inputPath, outputPath)`.
    2. Generate a `web_preview` version of every uploaded image with the watermark.
- **acceptance-criteria**:
    - A watermarked version of every photo is stored in B2 alongside the original.

### **Task 6: Conditional Media Access Logic (The "Swap")**
- **suggested-assignee**: Backend
- **priority**: High
- **consideration**: This is the financial bridge. Clients see the watermark until they pay.
- **description**: Update the `GetMediaURL` logic to serve the watermarked version if the project/milestone is not yet `completed`.
- **requirements**:
    1. Check milestone status for the project.
    2. If not paid: Return signed URL for the `web_preview` (watermarked).
    3. If paid: Return signed URL for the `orig` (high-res).
- **acceptance-criteria**:
    - A guest client sees watermarked images on an unfunded project, and high-res images immediately after funding.

### **Task 10: "Download All" Zip Generator**
- **suggested-assignee**: Senior
- **priority**: Medium
- **consideration**: Generating a 5GB zip on-the-fly will kill your server. Use a streaming approach or pre-generate it.
- **description**: A service that packages all project media into a single ZIP for the client.
- **requirements**:
    1. Implementation of `StreamProjectZip(projectID)`.
    2. Ensure only `orig` (high-res) files are included.
- **acceptance-criteria**:
    - Client clicks "Download All" and receives a zip file containing their high-res assets.

### **Task 11: Support for High-Resolution Video Transcoding**
- **suggested-assignee**: Senior
- **priority**: Low
- **consideration**: Videos are too large for standard resizing. Use FFmpeg wrappers.
- **description**: Generate low-bitrate "Preview" versions of uploaded MP4/MOV files.
- **requirements**:
    1. Trigger FFmpeg on the worker when `mime_type` starts with `video`.
    2. Store a 720p `.mp4` as the preview.
- **acceptance-criteria**:
    - A 4K video upload results in a fast-loading 720p preview in the gallery.

### **Task 12: Smart Cache-Control Implementation**
- **suggested-assignee**: Backend
- **priority**: Medium
- **consideration**: This reduces "Class B" transaction costs on B2 by preventing redundant URL signing.
- **description**: Set long-lived `Cache-Control` headers when generating signed URLs.
- **requirements**:
    1. Set `max-age=86400` (24 hours) for signed media URLs.
- **acceptance-criteria**:
    - Browser "Network" tab shows media files are served from disk cache on page refresh.
