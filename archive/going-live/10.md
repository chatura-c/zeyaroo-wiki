Topic 10: **Legal Infrastructure & Dispute Management**.
This module focuses on creating the "Legal Shield" discussed. It formalizes the relationship between the platform and its users, establishes the rules for escrow disputes, and ensures compliance with global privacy standards.

---

### **Task 1: Centralized Legal Document Hosting**
- **suggested-assignee**: Frontend
- **priority**: High
- **consideration**: Use Markdown for these pages so non-technical team members can update them easily via a simple file change.
- **description**: Create dedicated, public routes for Terms of Use and Privacy Policy.
- **requirements**:
    1. Create `/terms` and `/privacy` routes.
    2. Add these links to the footer of all public pages (Galleries, Proposals, Landing).
- **acceptance-criteria**:
    - Users can access the legal text from any screen without logging in.

---

### **Task 2: Mandatory "Agreement to Terms" on Registration**
- **suggested-assignee**: Fullstack
- **priority**: High
- **consideration**: We must store the timestamp and the version of the terms agreed to for audit trails.
- **description**: Prevent users from creating an account without explicitly agreeing to the platform's rules.
- **requirements**:
    1. Add a checkbox to `Register.tsx`: "I agree to the Terms of Use and Privacy Policy."
    2. Backend: Add `AgreedToTermsAt` and `TermsVersion` to the User model.
- **acceptance-criteria**:
    - The "Create Account" button is disabled until the checkbox is ticked.

---

### **Task 3: "Review & Quality Confirmation" Step for Clients**
- **suggested-assignee**: Frontend
- **priority**: High
- **consideration**: This acts as "In-App Evidence" that the client was happy with the watermarked work before releasing funds.
- **description**: Add a mandatory confirmation step before the final milestone release.
- **requirements**:
    1. In the "Release Payment" modal, add text: "I confirm that I have reviewed the watermarked previews and they meet the project requirements."
- **acceptance-criteria**:
    - The final payment release cannot be triggered without checking this box.

---

### **Task 4: Implement "Standard Service Agreement" (SSA) for Proposals**
- **suggested-assignee**: Backend
- **priority**: Medium
- **consideration**: This protects the platform by defining the legal contract *between* the creator and client.
- **description**: Automatically attach a standard contract template to every proposal sent through the platform.
- **requirements**:
    1. Create a default SSA text in the backend config.
    2. Include a link to the "Zeyaroo Standard Project Contract" on every public proposal page.
- **acceptance-criteria**:
    - Every proposal has a visible section stating: "Work governed by the Zeyaroo Standard Service Agreement."

---

### **Task 5: Formalized "Dispute Case" Model**
- **suggested-assignee**: Backend
- **priority**: High
- **consideration**: A dispute shouldn't just be a "status"; it needs to be an object with its own history.
- **description**: Create a database structure to track the progress of a disputed milestone.
- **requirements**:
    1. Create `models.Dispute`.
    2. Fields: `id`, `milestone_id`, `initiator_id`, `reason`, `status` (open, under_review, resolved), `resolution_type` (refund/release), `admin_id`.
- **acceptance-criteria**:
    - When a user clicks "Dispute" in the UI, a record is created in this table.

---

### **Task 6: Dispute "Evidence" Upload UI**
- **suggested-assignee**: Fullstack
- **priority**: Medium
- **consideration**: Both parties should be able to upload screenshots or logs to prove their case.
- **description**: Interface for users to submit proof when a milestone is in the `disputed` state.
- **requirements**:
    1. A "Dispute Center" view accessible from the project milestones tab.
    2. File upload field for "Evidence" (limit to 3 files/images).
- **acceptance-criteria**:
    - Creator and Client can see a shared "Dispute Evidence" log.

---

### **Task 7: Implement "Dispute Review Fee" Logic**
- **suggested-assignee**: Backend
- **priority**: Low
- **consideration**: This is an economic barrier to prevent "frivolous" disputes.
- **description**: Charge a small non-refundable fee (e.g., $15 or Rs. 2,000) to open a formal dispute.
- **requirements**:
    1. Before creating a `models.Dispute`, trigger a DodoPayments/PayHere checkout session.
    2. Only create the dispute record once payment is successful.
- **acceptance-criteria**:
    - The "Open Dispute" flow requires a payment transaction.

---

### **Task 8: "Right to be Forgotten" (Account Deletion) Service**
- **suggested-assignee**: Backend
- **priority**: Medium
- **consideration**: Ensure all media on Backblaze is also deleted to comply with GDPR.
- **description**: Provide a way for users to permanently delete their account and data.
- **requirements**:
    1. Endpoint: `DELETE /api/v1/users/me`.
    2. Logic: Delete user record -> Delete all projects -> Trigger B2 file deletion for all media keys.
- **acceptance-criteria**:
    - After deletion, no trace of the user's files remains on the server or storage.
