Topic 3: **Zero-Friction Lifecycle (Guest Mode & Project Claiming)**. 
The objective of this module is to allow clients to interact with proposals, pay milestones, and view media without ever creating a password until the very end of the project.

---

### **Task 1: Generate Secure Public Access Tokens for Proposals**
- **suggested-assignee**: Backend
- **priority**: High
- **consideration**: Use a cryptographically secure random string (UUID or similar). This token should be unique per proposal.
- **description**: Create a `access_token` field in the Proposal model to allow unauthenticated access via a "Magic Link."
- **requirements**:
    1. Add `AccessToken` (string) to `models.Proposal`.
    2. Auto-generate this token upon proposal creation.
    3. Create a new endpoint `GET /api/v1/public/proposals/:token`.
- **acceptance-criteria**:
    - The endpoint returns basic proposal and inquiry data without requiring an `Authorization` header.

---

### **Task 2: Implementation of Email-OTP Request Logic**
- **suggested-assignee**: Backend
- **priority**: High
- **consideration**: To prevent spam, limit a single email to 3 OTP requests per 10 minutes.
- **description**: Before a guest can "Accept" or "Pay," we must verify their email. Since they don't have a password, we will use a One-Time Password (OTP).
- **requirements**:
    1. Create a `verification_codes` table (email, code, expires_at).
    2. Endpoint: `POST /api/v1/auth/guest/request-otp` (takes email).
    3. Trigger a Resend email with the 6-digit code.
- **acceptance-criteria**:
    - A code is generated in the DB and an email is sent to the provided address.

---

### **Task 3: Guest-to-JWT Authentication Handler**
- **suggested-assignee**: Backend
- **priority**: High
- **consideration**: The JWT issued to a guest should have a specific scope `role: guest` and only allow access to resources associated with their email.
- **description**: Endpoint to verify the OTP and return a temporary JWT.
- **requirements**:
    1. Endpoint: `POST /api/v1/auth/guest/verify-otp`.
    2. Validate code; if correct, issue a JWT containing the `email` and `role: guest`.
- **acceptance-criteria**:
    - Providing the correct code returns a valid JWT that the frontend can use for subsequent "Guest" actions.

---

### **Task 4: Public Proposal Viewing UI**
- **suggested-assignee**: Frontend
- **priority**: High
- **consideration**: This is the first thing a client sees. It must look highly professional.
- **description**: Build a route `/p/:token` that renders the proposal details for unauthenticated users.
- **requirements**:
    1. Create a `PublicProposal.tsx` page.
    2. If the API returns a `401`, show the "Enter Email to View" OTP modal.
    3. Display deliverables, milestones, and total amount clearly.
- **acceptance-criteria**:
    - A user with a link can see the proposal data after a simple email verification.

---

### **Task 5: Support Guest Email in Milestone Payouts**
- **suggested-assignee**: Backend
- **priority**: Medium
- **consideration**: In `models.EscrowTransaction`, `client_id` will be null for guests, so ensure `guest_email` is captured instead.
- **description**: Update the payment logic to handle users who do not have a `user_id` yet.
- **requirements**:
    1. Update the `FundMilestone` service to accept `guest_email`.
    2. Ensure the DodoPayments metadata includes the email if `user_id` is missing.
- **acceptance-criteria**:
    - A guest can complete a checkout, and the milestone status updates to `funded`.

---

### **Task 6: The "Project Hand-Over" State Trigger**
- **suggested-assignee**: Backend
- **priority**: Medium
- **consideration**: The hand-over shouldn't be possible until the "Final Milestone" is paid.
- **description**: Create a logic check that flags a project as "Ready for Claiming."
- **requirements**:
    1. In `internal/services/project.go`, add `IsReadyForHandover(projectID)` logic.
    2. The project is ready only if all milestones are `completed`.
- **acceptance-criteria**:
    - Frontend receives a `can_be_claimed: true` flag in the project metadata.

### **Task 7: "Claim Your Account" Invitation UI**
- **suggested-assignee**: Frontend
- **priority**: High
- **consideration**: This is your "Conversion Hook." Make it feel like they are "Protecting" their investment.
- **description**: When a guest views their completed project, show a prominent banner to create a full account.
- **requirements**:
    1. Component: `ClaimProjectBanner.tsx`.
    2. Visual: "Your photos are ready. Create a password to claim this project and save it to your private vault."
- **acceptance-criteria**:
    - Banner appears only for guest-role users on completed projects.

---

### **Task 8: Technical Project Claim Service**
- **suggested-assignee**: Senior
- **priority**: High
- **consideration**: This is an ownership transfer. We must swap the `owner_id` from the Creator to the newly created Client ID.
- **description**: The backend service that creates a real user from a guest email and moves project ownership.
- **requirements**:
    1. Endpoint: `POST /api/v1/projects/:id/claim` (requires a `password` in body).
    2. Logic: Create `User` -> Change `Project.owner_id` -> Delete guest session.
    3. Grant the new user a "Starter" storage tier automatically.
- **acceptance-criteria**:
    - After claiming, the creator no longer pays for the storage of that project; the client does.

---

### **Task 9: Soft-Lock Funded Guest Projects**
- **suggested-assignee**: Backend
- **priority**: High
- **consideration**: This protects the client's money. If they paid Zeyaroo, the creator shouldn't be able to "take the money and run" by deleting the project.
- **description**: Prevent creators from deleting projects that have `funded` milestones until a resolution is reached.
- **requirements**:
    1. Add a check in `DeleteProject` service.
    2. If any milestone is `funded` or `submitted`, return `403 Forbidden`.
- **acceptance-criteria**:
    - Creator receives a "Cannot delete project with active funds" error.

---

### **Task 10: Magic Link "Copy to Clipboard" Tool**
- **suggested-assignee**: Junior
- **priority**: Low
- **consideration**: Use a "Share" icon and provide visual feedback ("Link Copied!").
- **description**: Give the creator a button in their dashboard to quickly copy the public link for a proposal.
- **requirements**:
    1. Add a "Copy Public Link" button to `ProposalDetail.tsx`.
    2. Generate the URL using `window.location.origin + '/p/' + proposal.token`.
- **acceptance-criteria**:
    - Creator can click one button and get the link ready to paste into WhatsApp/Email.

### **Task 11: Auto-Provision "My Deliveries" Project for Clients**
- **suggested-assignee**: Backend
- **priority**: Medium
- **consideration**: This is the "Delivery Folder" where claimed projects land.
- **description**: When a client claims their first project, ensure a system project named "My Deliveries" exists to hold it.
- **requirements**:
    1. In the `ClaimProject` service, check if the client has a system project.
    2. If not, call `CreateProject` with `is_system: true`.
- **acceptance-criteria**:
